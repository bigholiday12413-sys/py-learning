# Lv18 - ローカルLLM（Ollama）で構造化抽出

## テーマ

ローカルLLM（Ollama + qwen3:8b）を使って、日本語の営業日報テキストから
**5つの項目（商材名・顧客名・競合情報・商談結果・顧客の反応）を JSON で抽出**する。

これまで学んだ `requests`（Lv04, Lv17）の知識だけで LLM を叩けることを体感し、
最終的には **約200件の日報を一括処理 → 100件を目視検証して一致率8割以上** を目指す。

処理の流れ:

```
sample_reports.csv（日報の山）
        │
        ▼
batch_extract.py ──→ Ollama (qwen3:8b) に1件ずつ投げる
        │
        ▼
results.jsonl（抽出結果を1行ずつ追記保存）
        │
        ▼
verify.py + answers.csv（手作業で作った正解データ）
        │
        ▼
一致率レポート + 不一致リストCSV（目視確認用）
```

## Ollamaとは

Ollama は、**自分のPCの中で LLM を動かすためのツール**。
ChatGPT のような API を `http://localhost:11434` でローカルに立ててくれる。

なぜクラウドの LLM（ChatGPT / Claude 等の API）ではなくローカルLLMを使うのか:

| 観点 | ローカルLLM (Ollama) | クラウドLLM API |
|------|---------------------|----------------|
| **社内データの扱い** | 外部に一切送信されない（日報＝顧客名入りの機密データでも安心） | 外部サーバーに送信される（社内規定NGの場合が多い） |
| **料金** | 無料（電気代のみ） | 従量課金（200件×毎日だと積み上がる） |
| **ネットワーク** | オフラインで動く | インターネット必須 |
| **精度** | 8Bクラスはやや劣る（→だから検証が大事） | 最高クラス |
| **速度** | PCのGPU/メモリ次第 | 安定して速い |

「顧客名・商談内容が入った日報を外に出せない」という業務要件がある場合、
ローカルLLMはほぼ唯一の選択肢になる。精度が落ちる分を **検証（verify.py）** で担保する、
というのが今回の設計思想。

## 動かし方

```bash
# 1. Ollama をインストール（初回のみ）
#    https://ollama.com/download から Windows インストーラーをダウンロードして実行。
#    インストール後は Ollama がバックグラウンドで常駐し、
#    http://localhost:11434 で API サーバーが自動的に起動する。
#    （手動で `ollama serve` を打つ必要は基本ない。タスクトレイのラマのアイコンが目印）

# 2. モデルをダウンロード（初回のみ・約5GBなので時間がかかる）
ollama pull qwen3:8b

# 3. 動作確認（対話モードで話しかけてみる。/bye で終了）
ollama run qwen3:8b

# 4. venv を作成・有効化（他のレベルと同じ）
python -m venv venv
.\venv\Scripts\Activate.ps1

# 5. 依存パッケージをインストール
pip install -r requirements.txt

# 6. まず1件だけ抽出してみる（main.py が入門）
python main.py

# 7. CSV の10件を一括処理
python batch_extract.py

# 8. 正解データ（answers.csv）を手作業で作ってから精度検証
python verify.py
```

## 学べること

| 今回学ぶこと | これまでの知識との対応 |
|--------------|----------------------|
| `POST /api/chat` で LLM を呼ぶ | Lv17 の Kintone REST API と同じ `requests.post()` |
| system / user プロンプトの役割分担 | 「役割設定」と「実データ」を分ける設計 |
| `format` パラメータ（JSONスキーマ指定） | LLM の出力を型で縛る＝構造化出力 |
| `temperature=0` | 出力を毎回同じにする（抽出タスクの鉄則） |
| `json.loads()` + 例外処理 | Lv02 の JSON 処理 + try/except |
| JSONL への逐次追記 | 途中クラッシュしても進捗が消えない設計 |
| レジューム（再開）処理 | RPA でもおなじみの「途中から再実行」 |
| 正解データとの突き合わせ | LLM の出力は必ず検証する、という運用姿勢 |

## 読む順番

1. この README を読む（特に「ハマりポイント」を先に）
2. Ollama をインストールして `ollama pull qwen3:8b` を済ませる
3. `main.py` を上から読む → `python main.py` で1件抽出を体験
4. `sample_reports.csv` を開いて、日報にどんな「揺れ」があるか眺める
5. `batch_extract.py` を読む → 実行 → `results.jsonl` を開いてみる
6. `results.jsonl` を見ながら正解データ `answers.csv` を手作業で作る
7. `verify.py` を実行して一致率を確認 → 不一致CSVを目視レビュー
8. 改造課題に挑戦する

## ハマりポイント

### VRAM / メモリ要件

- qwen3:8b は実行に **約8GB のメモリ（GPUならVRAM）** が必要。
- GPU の VRAM が足りない場合は自動的に CPU + メインメモリで動くが、**かなり遅くなる**
  （1件数秒 → 数十秒〜）。200件処理の前に1件あたりの時間を必ず計測すること。
- メモリが厳しいPCでは `qwen3:4b`（約4GB）を試す。精度は落ちるので verify.py で要確認。
- `ollama ps` で今ロードされているモデルと GPU/CPU の割合が確認できる。

### 初回ロードが遅い

- モデルは**最初のリクエスト時にメモリへロード**される。初回だけ数十秒待たされるのは正常。
- 一度ロードされると既定で約5分間メモリに常駐するので、2件目以降は速い。
- バッチ処理の「1件目だけ異常に遅い」のはこのため。タイム計測時は1件目を分けて考える。

### qwen3 の thinking（思考）モード

- qwen3 は回答の前に「考え中」のテキスト（thinking）を出す癖がある。
- 抽出タスクでは不要でトークンの無駄＝遅くなるので、**API の `think: false` で無効化**する
  （main.py 参照）。プロンプトに `/no_think` と書く方法もある。

### 日本語プロンプトのコツ

- **項目名を曖昧にしない**。「競合」だけでなく「競合情報（他社製品名や他社の動き。なければ null）」
  のように定義と null 条件まで書く。
- **出力形式を必ず指定**し、さらに `format` パラメータ（JSONスキーマ）で機械的にも縛る。
  「プロンプトでお願い + スキーマで強制」の二段構えが安定する。
- **本文にない情報は null にせよ**と明示しないと、LLM は気を利かせて捏造（ハルシネーション）する。
- 敬体・常体は精度に大差ないが、指示は**箇条書きで短く**書くほうが安定する。

### その他

- `requests.exceptions.ConnectionError` が出たら Ollama が起動していない。
  タスクトレイを確認するか、ターミナルで `ollama serve` を実行する。
- `format` でスキーマを指定しても、ごく稀に壊れた JSON が返ることがある。
  `json.loads()` は必ず try/except で包む（main.py 参照）。
