# Lv13 - pytest入門：壊れていないことを自動で確認する【発展編】

## テーマ

ツールを改造するたびに「全部手で動かして確認」するのは限界がある。
**テストコード**を書いておけば、コマンド1つで「壊れていないこと」を数秒で確認できる。

- 改造課題に安心して取り組める（壊したら即分かる）
- 「前は動いていたのに」を撲滅する
- テストしやすいコードを書く習慣は、そのまま良い設計の習慣になる

## 動かし方

```bash
cd lv13-pytest
python -m venv venv
.\venv\Scripts\Activate.ps1     # Windows / macOSは source venv/bin/activate
pip install -r requirements.txt

pytest -v        # 全テストを実行（-v で1件ずつ名前を表示）
```

成功すると全テストに `PASSED` が並ぶ。
試しに `parsers.py` をわざと壊して（例: `rating_map` の `"Three": 3` を `4` に変更）
もう一度 `pytest` を実行すると、**どのテストが・どう失敗したか**を教えてくれる。

## pytest の基本ルール

| ルール | 内容 |
|--------|------|
| ファイル名 | `test_*.py` が自動で発見される |
| 関数名 | `test_` で始まる関数が1つずつ実行される |
| 判定 | `assert 式` が全部成立すれば PASS |
| 失敗時 | 期待値と実際の値を並べて表示してくれる |

```python
def test_parse_price_basic():
    assert parse_price("£51.77") == 51.77
```

これだけでテストとして成立する。特別な作法はほぼ不要。

## 学べること

| トピック | ひとことで言うと |
|---------|----------------|
| `assert` | 「こうなるはず」を書く。テストの基本単位 |
| `pytest.raises` | 「エラーになるはず」をテストする（異常系） |
| `@pytest.mark.parametrize` | 同じテストを複数データで回す（表形式のテスト管理） |
| `@pytest.fixture` | テストの準備データを共通化。テストごとに作り直される |
| 境界値テスト | バグは「ちょうど境界」に潜む（max_length ちょうど / ±1） |
| ゼロ件テスト | 空リスト・空文字列の挙動も必ず確認する |

## テストしやすいコードの鉄則

`parsers.py` は Lv09 のスクレイパーから「文字列の加工ロジック」だけを切り出したもの。

```
【テストしにくい】                 【テストしやすい】
def scrape(page):                  def scrape(page):
    text = page.locator(...)           text = page.locator(...)
    # 取得と加工が混ざっている          return parse_price(text)  # 加工は関数へ
    price = float(text.strip()
        .replace("£", ""))         def parse_price(text):        # ← 純粋な関数。
    ...                                ...                        #    ブラウザ不要でテスト可能
```

**「ブラウザ操作・通信」と「データ加工」を関数レベルで分離する**と、
加工ロジックはブラウザを起動せずに一瞬でテストできる。
これはテストのためだけでなく、コードの読みやすさ・再利用性も上げる設計術。

## 補足: Playwright のテスト

pytest には Playwright 公式プラグイン（`pytest-playwright`）もあり、
「ブラウザ操作そのもの」をテストとして書ける（Webサイトの E2E テストという分野）。
まずこのレベルで pytest の基本を固めてから公式ドキュメントに進むとスムーズ。
