# 補講B - セレクタの調べ方：ブラウザ開発者ツール入門

## テーマ

Lv04〜Lv09 では「CSSセレクタで要素を指定する」場面が何度も出てくるが、
実際のサイトで **「このボタンのセレクタは何か？」を自力で調べる方法** がこの補講のテーマ。

スクレイピング・自動操作の実務は、コードを書く時間より
「セレクタを調べて試す時間」の方が長いことも多い。ここが実務力の分かれ目。

**読むタイミング: Lv04 の途中（セレクタが出てきたあたり）がおすすめ。**

---

## 1. 開発者ツール（DevTools）を開く

どのブラウザ（Chrome / Edge / Firefox）でもほぼ同じ:

- **F12** キー、または右クリック → **「検証」**（Inspect）
- 画面の右か下に、HTMLがツリー表示されたパネルが出る（Elements / 要素 タブ）

### 一番よく使う操作: 「この要素はHTMLのどこ？」

1. ページ上の調べたい場所（ボタン・タイトルなど）を **右クリック → 検証**
2. Elements パネルで、その要素のHTMLがハイライトされる
3. そのタグの `id` / `class` / `name` などの属性を見る

例: books.toscrape.com の本のタイトルを検証すると、こう見える:

```html
<article class="product_pod">
    <h3>
        <a href="catalogue/..." title="A Light in the Attic">A Light in ...</a>
    </h3>
    <p class="price_color">£51.77</p>
</article>
```

ここから Lv04〜06 で使ったセレクタが導ける:

- 本の1件分のコンテナ → `article.product_pod`
- タイトルのリンク → `article.product_pod h3 a`
- 価格 → `.price_color`

## 2. CSSセレクタの文法 早見表

| 書き方 | 意味 | 例 |
|--------|------|-----|
| `div` | タグ名で指定 | すべての `<div>` |
| `.price` | class で指定 | `class="price"` を持つ要素 |
| `#login` | id で指定 | `id="login"` の要素（idはページ内で一意） |
| `div.item` | タグ + class | `class="item"` の `<div>` だけ |
| `[name="q"]` | 属性で指定 | `name="q"` を持つ要素 |
| `input[type="submit"]` | タグ + 属性 | 送信ボタン |
| `A B` | 子孫（間にスペース） | A の内側にある B（何階層下でも） |
| `A > B` | 直下の子 | A の1階層下の B だけ |
| `A, B` | どちらか | A または B に一致する要素 |
| `li:nth-child(2)` | n番目 | 2番目の `<li>` |

## 3. セレクタをその場でテストする（Ctrl+F 検索）

書いたセレクタが正しいかは、**コードを実行する前に** DevTools 上で確認できる。

1. Elements パネルをクリックしてから **Ctrl + F**（検索バーが出る）
2. セレクタを入力する（例: `article.product_pod h3 a`）
3. **ヒット件数** が表示され、一致箇所が順番にハイライトされる

チェックポイント:

- 件数が **0** → セレクタが間違っている（タイプミス、構造の勘違い）
- 件数が **想定より多い** → 絞り込みが足りない（余計な要素まで拾ってしまう）
- 件数が **想定通り** → そのまま Python コードに使ってOK

この「Ctrl+F で件数確認 → コードに貼る」の往復が、セレクタ調べの基本サイクル。

## 4. 「Copy selector」を鵜呑みにしない

DevTools では要素を右クリック → Copy → **Copy selector** で
セレクタを自動生成できるが、そのまま使うのはおすすめしない。

自動生成の例:

```
#default > div > div > div > div > section > div:nth-child(2) > ol > li:nth-child(1) > article > h3 > a
```

これは「今この瞬間の階層構造」に完全に依存していて、
サイトのレイアウトが少し変わっただけで壊れる。**壊れにくいセレクタ**を自分で選ぶこと:

### 良いセレクタの優先順位

1. **`#id`** … ページ内で一意。最も安定
2. **`[name="..."]`** … フォーム部品はこれが安定（サーバー送信に使われる名前なので変わりにくい）
3. **意味のある class** … `.price_color` のような「役割を表す」class
4. **テキストで指定**（Playwright限定）… `page.get_by_text("ログイン")` / `page.get_by_role("button", name="送信")`
5. ❌ `div:nth-child(3) > div > span` のような構造依存の長いチェーン

※ `css-1x2y3z` のようなランダム文字列っぽい class は、
ビルドのたびに変わる自動生成名であることが多いので避ける。

## 5. Playwright codegen — 操作を記録してコードを自動生成

Playwright には **操作を録画して Python コードに変換してくれる** ツールが付属している。
セレクタ調べが一気に楽になる、実務でも定番の機能。

```powershell
# venv を有効化した状態で（Lv05 以降の環境ならすぐ動く）
playwright codegen https://books.toscrape.com/
```

実行すると:

1. ブラウザと「Playwright Inspector」ウィンドウが開く
2. ブラウザ上でクリック・入力すると、**対応する Python コードがリアルタイムで生成される**
3. Inspector 上部で言語を `Python`（sync）に切り替えられる
4. 生成コードをコピーして、自分のスクリプトに貼り付けて調整する

注意点:

- 生成コードは「たたき台」。そのままだと待機処理やエラー処理が足りないので、
  Lv06 で学んだ待機戦略・try/except を自分で足すこと
- codegen が選んだセレクタも万能ではない。上の「良いセレクタの優先順位」で見直す
